## Slice 1 — Bare daemon process: starts, idles, stops cleanly

Goal: `record daemon start` launches a long-lived `python -m record.daemon` process that claims a PID file at `~/Library/Application Support/record/daemon.pid`, writes structured logs to `~/record/logs/daemon.log` (hard-coded for now; slice 3 makes it configurable), and exits cleanly on SIGTERM. `record daemon stop` signals it. `record daemon restart` chains stop→start. The existing `record start` / `record stop` path is untouched.

## Slice 4 — Swift `--daemon` mode: one long-lived capture process across many start/stop cycles

Goal: the Swift binary gains a single `--daemon` flag; under it, the process emits `ready` and idles waiting for commands instead of capturing immediately, resets `capture = nil` after `stopped` instead of `exit()`-ing, and only exits on `shutdown` / EOF / SIGTERM. The daemon spawns one long-lived `record-capture --daemon` child at startup and reuses it for every capture. The one-shot path (no `--daemon`) is unchanged so the existing 001/002 integration suite continues to pass.
